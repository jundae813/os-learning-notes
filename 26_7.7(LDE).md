# LDE: 제한적 직접 실행 원리 (OSTEP ch6)

CPU를 나누어 씀으로써 가상화를 구현하는데 다음과 같은 문제가 생김.
-> 성능저하 (소프트웨어가 명령 하나하나를 대신 실행하면(에뮬레이션) 하드웨어 직접 실행보다 수십~수백 배 느림)

왜 제한적인가?

특권/격리 측면: 프로그램을 CPU에 완전히 풀어놓으면 특권 명령(하드웨어 직접 조작, 다른 프로세스 메모리 접근)을 그냥 실행 
→ 격리·권한 경계 붕괴.
CPU 점유 측면: 프로그램이 CPU를 자발적으로 안 놓으면 OS가 영영 제어를 못 되찾아 
→ 다른 프로세스가 굶고 시스템이 멈춤.

그래서 직접 실행을 통해 해결함

####직접 실행 순서

1.프로세스 목록에 항목 생성
2.메모리 할당
3.메모리에 프로그램 탑재
4.argc/argv를 위한 스택 셋업
5.레지스터 내용 삭제
6. call main 실행

허나 1. 프로그램 직접 실행시 운영체제가 원치 않는 일을 하지 않는다는 것을 어떻게 보장하며 2. 프로세스 실행 시, 운영체제는 어떻게 프로그램의 실행을 중단하고 전환을 어떻게 하는가.
2번은 시분할을 어떻게 구현하는가와도 연관이 됨.

####문제 및 해결

######1. 제한된 연산
빠르나 I/O입출력 또는 시스템 추가할당 요청시 어떻게 하는가 프로세스가 시스템에 대한 권한이 없기에 사용자모드와 커널 모드로 나누어 해결한다.
또한 제한 작업의 실행을 위해 시스템 콜을 제공하는데 이때 trap을 사용한다.(커널로 명령어 분기하는 동시에 특권 수준을 커널 모드로 상향 조절하는 것)
완료시 return -from -trap 명령어로 하향 조정한다.

중요) 하드웨어시 주의, rft명령어 실행 시, 사용자 프로세스로 제대로 리턴. x86은 PC, 플래그와 레지스터를 각 프로세스의 커널 스택에 저장, rft시 스택에서 팝, 사용자 모드 프로그램을 다시 시작함

Q. trap이 운영체제의 어디까지 실행할지 어떻게 아는가 -> 호출한 프로세서는 주소명시 X -> 커널은 부팅시 트랩 테이블을 만들고 통제한다.
이때 하드웨어에게 트랩 테이블 위치를 알려주는 건 특권 명령어다!

| 단계 | 주체 | 동작 | 모드 전환 |
|---|---|---|---|
| trap 진입 (시스템 콜 호출) | 하드웨어 | 레지스터·PC를 커널 스택에 저장, 트랩 핸들러로 분기 | 사용자 → 커널 (권한 상승) |
| trap 처리 (커널 작업) | OS | 시스템 콜 수행 | 커널 (전환 없음) |
| trap 복귀 (return-from-trap) | rft (하드웨어) | 커널 스택에서 레지스터 복원 | 커널 → 사용자 (권한 하향) |

######2. 프로세스 간 전환 (간단해야하나 까다로움)
CPU에서 프로세스 실행 = 운영체제의 본래 코드 자체가 실행 X (= 운영체제는 실행 X)
why? 1. 컴퓨터 성능저하를 막기 위해 OS가 프로세스에게 CPU를 내줌. 2. 한 코어는 한 순간에 한 모드(시용자 모드/커널 모드)에서 한 프로세스만 실행한다

그러면 어떻게 다시 CPU를 다시 획득하나?

1. 협조방식: 시스템 콜을 기다림(이때 운영체제가 프로세스의 합리적 행동을 신뢰한다는 조건하)
하지만 비우호적 프로세스가 안 놓으면 방법이 있나?
주로 yield 시스템 콜로 OS에게 넘기는데 비정상 또는 불법 시 trap으로 CPU를 획득해 프로세스를 종료시킴.

2. 선점형 방식

만약 프로세스가 무한 루프에 걸릴 경우 협조 방식에선 무한 루프 걸리면 재부팅밖에 답이 없다
그래서 선점형 방식 생김.
OS가 유저 프로세스 실행 전 하드웨어에 타이머 미리 설정
-> 타이머 인터럽트를 이용(수밀리초마다 인터럽트 발생시킴)
-> 발생시 이미 있는 OS 인터럽트 핸들러를 실행해서 제어를 획득
타이머를 설정하는 건 OS(부팅 때), 발동하는 건 하드웨어(유저 실행 중).

부팅 중에 운영체제는 타이머를 시작해 놓으니 제어가 돌아올 것을 알기에 안심하고 사용자 프로그램에 CPU를 넘긴다
하드웨어에선 인터럽트 진입 시 저장 → 나중에 rft로 복원 후 재시작

제어 되찾으면 실행 재개 여부는 스케줄러가 결정
만약 다른 프로세스로 전환시 문맥 교환 코드로 실행(이건 OS 고정 루틴)

문맥 교환이란?
운영체제가 해야하는 작업은 현재 실행 중인 프로세스 레지스터값 저장하고 실행될 건 복원하는 과정.(이땐 저수준 어셈블리어를 사용함)

따라서 두 번의 레지스터 저장/복원이 발생한다. (한 번은 타임 인터럽트(사용자 레지스터->커널 스택 (하드웨어)), 다른 한 번은 전환시(커널 레지스터->프로세스 구조체 (OS))

그러면 어떤 기준으로 프로세스를 전환하나?
-> SJF 최소작업 스케줄링: 짧은 작업을 먼저 실행하라.
허나 이 규칙을 실행하려면 OS가 지금 이 순간, 어느 작업이 짧을지를 알아야 함. 하지만 실제로는 알 수 없음.
why? 실제로 입력 값과 분기에 따라 실행되는 코드가 다르기 때문
따라서 프로그램이 CPU를 얼마 만에 스스로 놓느냐를 기준으로 과거 이력을 보고 우선순위를 되매김. (MLFQ)



만약 시스템 콜 처리 중 타이머 인터럽트 발생 시 또는 인터럽트 처리 중 인터럽트가 온다면?
-> 인터럽트 불능화! 인터럽트를 놓칠 수도 있음
단일 CPU에선 짧게 인터럽트를 끄는 걸로 충분하지만, 멀티프로세서에선 인터럽트 끄기만으로 부족해서 lock이 필요
