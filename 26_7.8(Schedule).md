# scheduler

워크로드: 일련의 프로세스들이 실행하는 상황

프로세스/작업에 대한 가정
1. 모든 작업은 같은 시간 동안 실행된다.
2. 모든 작업은 동시에 도착
3. 각 작업은 시작되면 완료될 때까지 실행
4. 모든 작업은 CPU에서 사용(입출력 수행 x)
5. 각 직업의 실행시간은 사전에 알려져 있음

워크로드 가정외에 정책 비교를 위해 스케줄링 평가 항목을 결정해야한다.

#### 1. 반환시간(turn around time)

Tt = T(completion)-T(arrival)
이때 동시에 도착한다 가정, 그러므로 T(arrival) = 0 Tt = Tc
성능 측면 평가

#### 2. 공정성(fairness)
예를 들어, 스케줄러는 성능을
극대화하기 위해 몇몇 작업의 실행을 중지하며, 결과적으로 공정성이 악화

성능과 공정성은 tradeoff

가장 기초적 알고리즘 FIFO/FCFS
다만, 긴 작업시간이 먼저 올 경우 반환시간이 나빠짐
짧은 시간 동안 CPU를 사용할 프로세스들이 CPU 오랫동안 사용하는 프로세스의 종료를 기다리는 현상
-> convoy effect

해결방안 -> SFJ : 가장 짧은 실행 시간을 가진 작업 먼저 실행
허나 작업 동시 도착이 비현실적
먼저 도착한 것이 긴 작업시간이라면 긴 작업 시간을 먼저 처리해야함.

해결방안 -> 선점형 스케줄러 : 다른 프로세스를 실행시키기 위해 현재 프로세스를 중단
선점은 타이머 인터럽트로 제어권을 강제 회수할 수 있기에 가능
SJF는 비선점이라 도착 순서에 따라 여전히 convoy가 생김 -> 선점을 붙인 STCF가 이를 해결
SJF + 선점기능 -> 최단 잔여시간 우선(STCF)/ 선점형최단 작업 우선(PSJF)

#### 3. 응답시간
시분할 등장으로 상호작용을 원할히 하기 위한 성능 요구

T(response) = T(firstrun)-T(arrival)
반환시간기준 훌륭한다면 응답시간과 상호작용 측면에서 나쁨.

해결방안 -> 라운드 로빈(RR) : 일정시간 실행 후, 실행 큐의 다음 작업으로 전환(일정시간: 타임슬라이스/ 스케줄링 퀀텀)

타임 슬라이스가 짧으면 짧을수록 응답시간은 좋아진다.
-> 하지만 바뀌면서 발생하는 문맥교환 비용이 전체 성능에 큰 영향을 미침.(문맥 교환 비용: 레지스터 복원/저장 + CPU 캐시 + TLB + 분기 예측 ....새 프로세스 것으로 다시 채워지는 간접 비용)
-> 문맥 교환 비용을 상쇄할 수 있을 만큼 길어야 함.

정리)
FIFO (기초) -> convoy문제
-> SJF (짧은 것 먼저) -> 동시도착 가정이 비현실적, 선점 없음
-> STCF (선점 추가) -> 반환시간엔 좋은데 응답시간이 나쁨
-> RR (응답시간용) -> RR은 반환시간이 나쁨 = 트레이드오프 
-> + 가정 5(실행시간 예지)가 거짓이라 SJF류 구현 불가
-> MLFQ (과거 관찰로 근사)
